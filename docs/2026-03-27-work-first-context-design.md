# Tasklog MCP Work-First Context Design

Date: 2026-03-27
Status: Proposed

## Summary

Tasklog MCP should move from a log-first model to a work-first model.

In the current model, the AI mainly recovers context from chronological logs. That works for short histories, but it degrades once a project has many unfinished threads, many related notes, or multiple design and implementation artifacts. The AI spends too much context resolving which thread matters, whether an old "open" entry is still active, and which notes belong together.

The new model keeps Markdown as the canonical storage format but introduces a stronger hierarchy:

- `project` is inferred automatically from the MCP launch path.
- `work` becomes the primary unit of discovery and execution.
- `log` becomes a session activity record, not a general note bucket.
- `note`, `design`, `plan`, and `spec` become first-class work artifacts.
- `active_work` becomes a convenience hint for the current session, not the source of truth.
- work docs are stored in one central location, with lightweight scope metadata instead of per-repo doc trees.

## Problems in the Current Model

### Log IDs are too expensive

Current UUID-shaped log ids are reliable but verbose. They cost context, are hard for humans to remember, and make related-entry references noisy.

The replacement should be short human-facing ids with a fixed 6-character base62 alphabet: `0-9`, `a-z`, and `A-Z`.

### Open-thread semantics are overloaded

Today an entry can be returned by `get_open_threads` because it is `WIP`, `Blocked`, or merely carries `next_steps`. This mixes "unfinished work" with "follow-up worth remembering" and makes the AI over-read old context.

### Logs are doing too many jobs

The current log entry is asked to be:

- a session summary
- a handoff record
- a backlog marker
- a place to stash future ideas

That makes the logbook useful, but it also causes ambiguity about whether the AI should append a log, amend an old one, or interpret a note as still-open work.

### Active work is unsafe if treated as truth

If the AI or user forgets to close or switch an `active_work`, future writes can drift into the wrong thread. This is especially risky when the user starts a new task in the same project without explicitly resetting context.

## Goals

- Reduce context spent on deciding which task the AI should look at first.
- Make cold-start recovery work even when many unfinished works exist.
- Keep Markdown readable and editable by humans outside the MCP.
- Let the AI create and query structured work artifacts without inventing ids or paths.
- Preserve a lightweight local workflow with minimal ceremony for small tasks.

## Non-Goals

- This design does not introduce a remote database.
- This design does not require multi-project coordination inside one MCP instance.
- This design does not eliminate chronological logs; it narrows their role.

## Core Model

### Project

`project` is derived automatically from the MCP launch path or configured project root.

Examples:

- `/Users/Lab/Desktop/WebWay/CodeWebway`
- `/Users/Lab/Desktop/WebWay/WebWayFleet`
- `/Users/Lab/Desktop/WebWay/tasklog-mcp`

The AI should not be asked to choose the current project during normal operation. Every tool should default to the inferred project, even when the workspace contains multiple repos.

### Work

`work` is the primary unit of discovery, resume, and execution inside a project.

A work represents one coherent piece of ongoing effort, such as:

- `login-handoff`
- `secure-launch-placeholder`
- `agent-session-cleanup`

Each work should have:

- `work_id`: short generated id, fixed at 6-character base62
- `title`: human-readable name
- `slug`: stable readable identifier
- `status`: `active`, `blocked`, `done`
- `start_dir`: the directory where the work started
- `scope_paths`: the directories this work may legitimately touch
- `created_at`
- `updated_at`
- `tags`
- optional `summary`

`work_id` should be generated by the server, not by the AI.

Short ids are a core part of the design, not an implementation detail. Human-facing ids for works, logs, and related references should use the same 6-character base62 format to reduce context cost.

The model should stay minimal. `scope_type`, `primary_repo`, and `related_repos` should not be stored as first-class fields when they can be derived from `start_dir` and `scope_paths`.

### Artifacts

Artifacts belong to a work. Not every work needs every artifact.

- `log`: session activity and handoff
- `note`: short observations, reminders, loose ideas, TODO fragments
- `design`: goals, tradeoffs, chosen approach
- `plan`: execution sequence, touched areas, test plan
- `spec`: precise contract, schema, state, rules, acceptance details

### Active Work

`active_work` is a session-level hint. It exists to reduce repetitive tool arguments, but it must not be treated as the source of truth for work state.

Rules:

- `active_work` may be absent.
- `active_work` may become stale.
- `active_work` should not silently override obvious changes in user intent.
- `active_work` should not auto-resume closed work unless explicitly reopened.

## Canonical Storage

Markdown remains canonical so humans can read and edit artifacts directly.

Work docs should be centralized in one predictable location, even when the surrounding workspace contains multiple repos or a monorepo subtree.

Suggested layout:

```text
<workspace_root>/
  .tasklog/
    works.json
    session-log.json
    session-log.md
  workdocs/
    <work_id>-<slug>/
      design.md
      plan.md
      spec.md
      notes.md
```

Example:

```text
/Users/Lab/Desktop/WebWay/workdocs/a91K2x-work-first-context/design.md
```

This central location makes discovery cheaper for the AI because it has one place to search. Scope disambiguation should come from the work metadata, not from the document path itself.

`workdocs/` should be the human-facing document root. `.tasklog/` remains the machine-facing state root.

The `.tasklog` directory can hold machine-friendly indexes and caches, but those indexes must be rebuildable from canonical Markdown plus work metadata. The AI should never depend on hidden state that humans cannot recover.

## Scope Model

This design should separate "what this work may involve" from "what this specific implementation pass will touch."

### Work Scope

At the work level:

- `start_dir` identifies where the work began
- `scope_paths` identifies which directories are in scope for the work overall

If `scope_paths` contains one directory, the work is effectively single-repo.

If `scope_paths` contains multiple directories, the work is cross-repo or cross-package, but that does not imply every implementation step must touch all of them.

### Plan Targeting

At the plan level:

- `target_paths` identifies the subset of `scope_paths` that this plan will actively implement

This distinction matters because many works are cross-repo in concept but only need one side implemented in a given pass.

Examples:

- Work scope: `CodeWebway` and `WebWayFleet`
- Plan A target: `CodeWebway` only
- Plan B target: `WebWayFleet` only

If `target_paths` is omitted when a plan is created, the tool may default it to all `scope_paths`, but the AI should be able to narrow that set explicitly.

## Artifact Semantics

### Log

Use `log` for session activity:

- what changed
- why it changed
- what outcome was produced
- what the next session should do, if the work is still in progress

The log is not the right place for general notes or evolving design rationale.

### Note

Use `note` for lightweight capture:

- reminders
- observations
- ideas
- small TODOs
- partial findings that are not yet a formal plan or spec

If a user says "note this down", the default destination should be a work note, not a session log.

### Design

Use `design` when the task needs:

- goals
- constraints
- tradeoffs
- considered options
- chosen direction

### Plan

Use `plan` when the task needs:

- implementation sequence
- touched files or modules
- target paths for this implementation pass
- test plan
- rollout or backout notes

### Spec

Use `spec` when the task needs precise behavior:

- API contract
- field definitions
- state transitions
- validation rules
- acceptance criteria

`spec` should not be folded into `design` by default. It serves a different purpose.

## Agent Burden Reduction

The design must explicitly reduce how much the AI has to infer or remember.

### 1. Project is automatic

The AI should not choose a project during normal use. Project context is inferred from launch location.

### 2. Work is resolved by tools, not by guesswork

The AI should not invent `work_id`s and should rarely type them manually. Tools should create, resume, and suggest works based on recent activity, current status, and the user's request.

### 3. Active work is advisory, not authoritative

If `active_work` is stale or mismatched, tools should surface ambiguity instead of silently writing to the wrong work.

### 4. Discovery should return summaries, not raw logs

Cold-start recovery should work through work summaries. The AI should not need to read many chronological entries just to decide where to start.

### 5. Artifact type should match user intent

The AI should not dump every piece of information into `log`. If the user asks to plan, design, specify, or note something, the corresponding artifact tool should exist and be easy to call.

### 6. Scope metadata should stay minimal

The AI should not have to process a large scope schema on every read. `start_dir` and `scope_paths` are enough for the work level, and `target_paths` is enough for plan-level implementation targeting. Richer labels should be derived by tools instead of stored everywhere.

## Cold-Start Flow

This flow is for an AI that starts with little or no context and the user says something like "help me look at unfinished work."

1. Call `get_active_context`.
2. If there is a fresh valid `active_work`, treat it as a hint, not an automatic commitment.
3. If no valid active work exists, call `list_works(status=open, compact=true)`.
4. Read a short work shortlist that includes:
   - `work_id`
   - `title`
   - `status`
   - `updated_at`
   - short scope label derived from `start_dir` and `scope_paths`
   - short last-summary
   - short next-step summary
   - artifact availability (`design`, `plan`, `spec`, `notes`)
5. Pick the best candidate based on user intent.
6. Call `resume_work`.
7. Call `read_work_context`.
8. Read only the needed artifacts before acting.

This changes recovery from `logs -> infer thread -> infer scope` to `works -> choose scope -> read targeted context`.

## Active Work Safety Rules

To prevent stale or forgotten active state from corrupting future sessions:

- A new MCP session should not blindly trust an old active work.
- `get_active_context` should return freshness information such as `fresh`, `stale`, or `invalid`.
- `append_session_log` should default to the active work only when confidence is high.
- If the active work is `done`, tools should require explicit reopen or new-work selection.
- If the AI starts touching files or topics that do not match the active work, the tool should return an ambiguity signal instead of writing automatically.

## Tool Design

### New Tools

#### `get_active_context`

Returns:

- inferred project root
- inferred project id
- current `active_work`, if any
- active-work freshness or confidence

Use at the beginning of a session.

#### `list_works`

Lists work items for the current project.

Expected filters:

- `status`
- `limit`
- `query`
- `compact`

This is the main discovery surface for cold starts.

#### `start_work`

Creates a new work, generates a 6-character base62 `work_id`, sets it active, and returns its summary and paths.

#### `resume_work`

Marks an existing work as the current active work. It should accept a `work_id` and optionally support keyword resolution.

#### `set_work_status`

Updates a work status: `active`, `blocked`, or `done`.

#### `read_work_context`

Returns a concise overview of one work:

- summary
- current status
- artifact paths
- recent logs
- pending next steps

This should be a compact, AI-friendly summary, not a raw dump.

#### `create_design_doc`

Creates or opens `design.md` for the active work using a standard template.

#### `create_plan_doc`

Creates or opens `plan.md` for the active work using a standard template.

When the work has multiple `scope_paths`, this tool should allow the AI to create a plan for only the relevant subset via `target_paths`.

#### `create_spec_doc`

Creates or opens `spec.md` for the active work using a standard template.

#### `append_work_note`

Appends a short note to `notes.md` for the active work.

This is the correct destination for "note this down" requests.

### Existing Tools To Keep, With Changed Defaults

#### `append_session_log`

Keep the tool, but:

- default to the active work when confidence is high
- fail with an ambiguity response when confidence is low
- stop acting like a general note bucket

#### `get_recent_logs`

Keep the tool, but default to the current work unless the caller explicitly asks for project-wide history.

#### `update_log_status`

Keep the tool for lifecycle maintenance.

#### `amend_log_metadata`

Keep the tool for lightweight corrections.

### Tool To Deprecate

#### `get_open_threads`

Deprecate this in favor of `list_works(status=open)`.

The current semantics are too ambiguous because "has next steps" is not the same thing as "this work is still open."

## File Templates

Every work artifact should be created with minimal frontmatter so both humans and tools can identify ownership and status without reading the entire file.

Example:

```md
---
work_id: a91K2x
title: Work-first context model
slug: work-first-context
status: active
start_dir: /Users/Lab/Desktop/WebWay/tasklog-mcp
scope_paths:
  - /Users/Lab/Desktop/WebWay/tasklog-mcp
created_at: 2026-03-27T00:00:00.000Z
updated_at: 2026-03-27T00:00:00.000Z
tags:
  - context
  - workflow
---
```

## Migration Strategy

### Phase 1

- Introduce work storage and work discovery.
- Keep existing logs intact.
- Make logs optionally reference `work_id`.

### Phase 2

- Add artifact creation tools for `design`, `plan`, `spec`, and `note`.
- Change agent guidance from log-first to work-first.

### Phase 3

- Deprecate open-thread recovery as the primary resume mechanism.
- Reframe logs as session activity only.

## Decisions

- Keep Markdown as canonical storage.
- Derive project automatically from MCP launch path.
- Use work-first discovery.
- Keep work docs centralized in one location.
- Keep `active_work` as a convenience layer only.
- Separate `note` from `log`.
- Keep `design`, `plan`, and `spec` distinct.
- Keep scope metadata lightweight with `start_dir` and `scope_paths`.
- Let plans narrow execution scope through `target_paths`.
- Use 6-character base62 ids for all human-facing work and log references.
